# AI와 함께한 해커톤 개발기

## PDEX Terminal — Hyperliquid 퍼프 DEX 트레이딩 터미널

---

## 1. 프로젝트의 시작: 사내 AI 해커톤

### 해커톤 배경

사내 AI 해커톤에 2인 팀으로 참가했다. 팀 구성은 **개발자 1명 + QA 1명**이다.

해커톤의 주제는 "AI를 활용한 제품 고도화"였고, 우리 팀은 처음부터 끝까지 AI와 함께하는 개발 프로세스 자체를 실험해보자는 방향으로 접근했다.

### 아이디어 도출 과정

ChatGPT와 여러 차례 대화를 주고받으며 아이디어를 좁혀나갔다.

```
1. 팀원 중 한 명은 블록체인 관련 업무 담당
2. 비트코인, 주식 등 투자 관심 많음
3. 실제로 팀이 사용할 수 있는 실용적인 도구
4. 해커톤 기간 내에 MVP까지 도달 가능한 범위
```

- 블록체인 + 트레이딩 + 개발 역량을 결합할 수 있는 프로젝트
- 실제로 팀이 사용할 수 있는 실용적인 도구
- 해커톤 기간 내 MVP 완성이 가능한 범위

최종적으로 **Hyperliquid DEX 트레이딩 터미널**을 만들기로 결정했다. AI가 포지션과 주문을 실시간으로 분석해주는 Copilot 기능이 핵심 차별점이다.

## 2. AI 기반 개발 프로세스 ⭐

> 이 프로젝트의 핵심은 **모든 단계를 AI와 대화하며 진행**했다는 점이다.
> 각 단계마다 산출물을 만들고 → 팀이 검토하고 → 피드백 후 다음 단계로 넘어가는 사이클을 반복했다.

### 전체 흐름도

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        AI 기반 개발 프로세스                            │
│                                                                         │
│  ① 아이디어    ② 요구사항    ③ 와이어프레임   ④ 설계      ⑤ 구현   │
│  (ChatGPT)  →  (Kiro)    →   (Kiro)     →  (Kiro)  →  (Kiro)         │
│      │            │             │             │           │             │
│      ▼            ▼             ▼             ▼           ▼             │
│   대화 로그    requirements   wireframe     design.md   소스 코드       │
│              .md            .html         api-spec    + 테스트          │
│                                           .md                           │
│                                                                         │
│  ⑥ 기능추가  ⑦ 브랜딩    ⑧ 문서화    ⑨ 디자인개선                  │
│  (Kiro)   →  (Gemini)  →  (Kiro)   →  (Claude)                        │
│                                                                         │
│         ◀──── 검토 & 피드백 (개발자 + QA) ────▶                        │
│         "빠진 부분 추가해줘"  "이름 바꿔줘"  "폴백 순서 변경해줘"       │
└─────────────────────────────────────────────────────────────────────────┘
```

### ⑨ 디자인 개선 (Claude)

구현과 문서화가 끝난 뒤, Claude에게 현재 디자인 트렌드와 개선점을 물어보는 방식으로 UI를 다듬었다.

```
QA → Claude: "이 와이어프레임 기반으로 현재 트레이딩 UI 트렌드와 개선점 제안해줘"
Claude → QA: 트렌드 분석 및 개선 포인트 제시
  - 다크 테마 컬러 시스템 (배경/액센트/위험 신호 역할 분리)

  - 실시간 데이터 강조를 위한 시각적 계층 구조 제안
QA: 레퍼런스 사이트/스크린샷 수집 → Claude에게 전달
QA → Claude: "이 레퍼런스처럼 수치 강조 방식 적용하면 어떨까?"
Claude → QA: 구체적인 적용 방안 제안
팀: ✅ 피드백 반영하여 UI 수정

### 핵심 포인트: 검토 사이클

> 모든 단계에서 **AI가 산출물을 만들고 → 팀이 검토하고 → 피드백**하는 사이클이 핵심이다.
> AI는 실행자, 팀은 의사결정자. 이 역할 분담이 명확해야 품질이 유지된다.

```

## 3. 주요 개발 과정 타임라인

### Phase 1: 프로젝트 구조 & 기본 구현

| 단계 | 내용 |
|------|------|
| 프로젝트 셋업 | 모노레포 구조 (web + was), TypeScript, Docker Compose |
| 데이터 레이어 | Hyperliquid API 클라이언트, Redis 캐시, MySQL 연동 |
| Rule Engine | Risk Calculator, S/R Calculator, Funding Analyzer, OI Analyzer, Liquidation Analyzer |
| AI Engine | LLM 클라이언트 (멀티 프로바이더), 프롬프트 빌더 |
| REST API | Express 라우터, 밸리데이션, 에러 핸들링 |
| 프론트엔드 | Next.js 15, 6개 컴포넌트 (TopBar, Portfolio, Market, AICopilot, Discover, BottomBar) |

### Phase 2: LLM 프로바이더 삽질기

AI 서비스 연동 과정에서 겪은 현실적인 문제들:

| 문제 | 상황 | 해결 |
|------|------|------|
| Gemini 429 에러 | 무료 티어 쿼터가 0 (사실상 사용 불가) | Groq를 1순위로 변경 |
| Groq 모델 폐기 | `llama-3.1-70b-versatile` 모델이 decommission됨 | `llama-3.3-70b-versatile`로 교체 |
| OpenAI 쿼터 초과 | 무료 크레딧 소진 | 3순위 폴백으로 유지 |

최종 폴백 체인: **Groq → Gemini → OpenAI**

교훈: 무료 LLM API는 언제든 제한될 수 있다. 멀티 프로바이더 폴백은 필수.

### Phase 3: 기능 고도화

| 기능 | 설명 |
|------|------|
| 오더 분석 | 전략 탐지, 체결 가능성, 주문 집중도, 포지션 영향 분석 파이프라인 |
| 시장 데이터 패칭 | 오더 분석 시 캔들(ATR), 펀딩 레이트, L2 오더북 추가 수집 |
| 전략 조언 (TP/SL) | LLM 기반 단기/중기 전략, TP/SL 라인, 시장 전망, 운용 팁 생성 |
| 코인 디스커버 | 시장 전체 데이터 기반 AI 코인 추천 (방향, TP/SL, 신뢰도) |
| UI 개선 | 리사이즈 가능한 BottomBar, Portal 기반 툴팁, 탭 레이아웃 최적화 |

### Phase 4: 문서화

| 문서 | 내용 |

| README.md (루트, web, was) | 프로젝트 소개, 빠른 시작, 환경 변수 |

### Phase 5: 브랜딩 & 비주얼 에셋

코드만 짜는 게 아니라, 프로젝트의 정체성을 만드는 작업도 QA와 AI가 함께 진행했다.

#### 프로젝트 네이밍

- ChatGPT와 대화하며 이름 후보를 브레인스토밍
- "Perp DEX Terminal" → 줄여서 **PDEX Terminal**로 확정
- 짧고 기억하기 쉬우면서 도메인(퍼프 DEX)을 담고 있는 이름

#### 로고 & 아이콘 생성

- **Gemini**를 활용하여 로고 시안 생성 (QA 주도)
- 프롬프트 예시: "미니멀한 트레이딩 터미널 로고, 다크 테마, 블루/퍼플 그라데이션, 차트 모티프"
- 여러 시안 중 팀이 함께 선택하고, 피드백을 반복하며 다듬음
- favicon, OG 이미지 등 필요한 사이즈별 에셋도 QA가 제작

#### 컬러 팔레트 & UI 톤

- Claude에게 현재 트레이딩 UI 트렌드를 물어보고 컬러 방향성 제안을 받음
- 레퍼런스 스크린샷을 전달하며 "이 느낌으로 적용하면 어떨까?" 방식으로 구체화
- 배경(#0f0f0f계열), 액센트(블루/그린), 위험 신호(레드/오렌지) 등 역할별 색상 체계 확립
- TailwindCSS 커스텀 컬러로 바로 적용

#### 브랜딩에서 배운 점

- AI에게 트렌드를 물어보는 것만으로도 방향성을 잡는 데 충분히 도움이 됨
- 레퍼런스를 함께 제공하면 추상적인 제안이 구체적인 적용 방안으로 바뀜
- 로고/이미지 생성은 프롬프트를 구체적으로 쓸수록 결과가 좋음

### Phase 6: 디자인 개선 (Claude 활용)

문서화와 브랜딩이 끝난 뒤, Claude에게 트렌드 분석과 레퍼런스 기반 개선을 요청하여 UI를 다듬었다:

| 단계 | 내용 |
|------|------|
| 트렌드 분석 | Claude에게 현재 트레이딩 UI 트렌드 및 개선점 제안 요청 |
| 레퍼런스 전달 | 팀이 수집한 디자인 레퍼런스를 Claude에게 공유, 적용 방안 논의 |
| UI 수정 | 제안 내용 반영하여 기존 UI 개선 |

## 4. AI와 협업한 방식의 특징

### 4.1 역할 분담: 개발자 vs QA vs AI

| 개발자 (기술 의사결정자) | QA (품질/UX 의사결정자) | AI (실행자) |
|--------------------------|-------------------------|-------------|
| 아이디어 제시 및 기술 방향 설정 | 와이어프레임 검토 및 UI 피드백 | 요구사항 문서 구조화 |
| 요구사항/설계 검토 및 피드백 | Claude에 트렌드/개선점 질의 | 와이어프레임 HTML 생성 |
| 기술적 판단 (LLM 선택, 인프라 구성) | 디자인 레퍼런스 수집 및 전달 | API 규격 문서 작성 |
| 빌드/실행 결과 확인 및 에러 전달 | 브랜딩 방향 결정 및 시안 선택 | 전체 코드 작성 (FE + BE) |
| UI 구현 가능 여부 피드백 | 최종 비주얼 검수 | 트렌드 분석 및 개선 제안, 로고/아이콘 생성 |
| 최종 기술 승인 | 최종 디자인 승인 | 문서 현행화 |

### 4.2 효과적이었던 패턴

1. **단계별 검토 사이클**: 한 번에 다 만들지 않고, 요구사항 → 검토 → 설계 → 검토 → 구현 → 검토 순서로 진행. 각 단계에서 방향을 잡아줌.

2. **단계 후 디자인 개선**: 구현과 문서화가 완료된 후 Claude와 함께 디자인 트렌드 분석 및 레퍼런스 기반 UI 개선을 진행. 완성된 코드 베이스 위에서 개선하므로 실제 반영이 빠름.

3. **자연어로 피드백**: "TP/SL 보다는 전략이라는 이름이 좋겠어", "하단 알림 컴포넌트 사이즈 조절할 수 있게 해줘" 같은 일상적인 표현으로 요청.

4. **에러 메시지 복붙**: 빌드 에러나 런타임 에러가 나면 그대로 복사해서 전달. AI가 원인 분석 후 수정.

5. **참고 자료 제공**: 다른 AI(Gemini)에게 받은 답변을 보여주며 "이런 느낌으로 만들어줘"라고 요청. AI가 해당 형식을 분석해서 구현.

### 4.3 주의할 점

- AI가 만든 코드를 무조건 신뢰하면 안 됨. 빌드/실행 확인은 필수.
- 문서와 코드가 따로 놀 수 있음. 구현 후 문서 현행화를 꼭 해야 함.
- LLM API 제한, 모델 폐기 등 외부 의존성 이슈는 직접 해결해야 함.
- 큰 기능은 한 번에 요청하지 말고 작은 단위로 나눠서 진행하는 게 효과적.
- Claude의 디자인 제안이 항상 구현 가능한 것은 아님. 개발자와 사전 협의가 필요.
