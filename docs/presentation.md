# AI와 함께한 프로젝트 개발기

## PDEX Terminal — Hyperliquid 퍼프 DEX 트레이딩 터미널

---

## 1. 프로젝트의 시작: ChatGPT와의 브레인스토밍

### 나는 누구인가

ChatGPT에게 나의 관심사와 상황을 전달하고, 1인 프로젝트로 할 수 있을 만한 것들을 추천해 달라고 요청했다.

```
1. 나는 소프트웨어 개발자임.
2. 블록체인 관련 업무 중
3. 비트코인, 주식 등에 관심 많음.
4. 비트코인, 이더리움 소유중.
5. 중학생 남자 아들 둘. 애들 교육에 관심.
6. 서울거주, 지하철로 출퇴근
7. 사이클, 마라톤, 등산 취미 있음
```

### 아이디어 도출 과정

ChatGPT와 여러 차례 대화를 주고받으며 아이디어를 좁혀나갔다.

- 블록체인 + 트레이딩 + 개발 역량을 결합할 수 있는 프로젝트
- 실제로 내가 사용할 수 있는 실용적인 도구
- 1인 개발로 MVP까지 도달 가능한 범위

최종적으로 **Hyperliquid DEX 트레이딩 터미널**을 만들기로 결정했다. AI가 포지션과 주문을 실시간으로 분석해주는 Copilot 기능이 핵심 차별점이다.

## 2. AI 기반 개발 프로세스 ⭐

> 이 프로젝트의 핵심은 **모든 단계를 AI와 대화하며 진행**했다는 점이다.
> 각 단계마다 산출물을 만들고 → 내가 검토하고 → 피드백 후 다음 단계로 넘어가는 사이클을 반복했다.

### 전체 흐름도

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        AI 기반 개발 프로세스                            │
│                                                                         │
│  ① 아이디어    ② 요구사항    ③ 와이어프레임   ④ 설계      ⑤ 구현   │
│  (ChatGPT)  →  (Kiro)    →   (Kiro)     →  (Kiro)  →  (Kiro)            │
│      │            │             │             │           │             │
│      ▼            ▼             ▼             ▼           ▼             │
│   대화 로그    requirements   wireframe     design.md   소스 코드       │
│              .md            .html         api-spec    + 테스트          │
│                                           .md                           │
│                                                                         │
│         ◀──── 검토 & 피드백 (나) ────▶                                  │
│         "빠진 부분 추가해줘"  "이름 바꿔줘"  "폴백 순서 변경해줘"       │
└─────────────────────────────────────────────────────────────────────────┘
```

### 단계별 상세

| # | 단계 | AI 도구 | 내가 한 일 | 산출물 |
|---|------|---------|-----------|--------|
| ① | 아이디어 도출 | **ChatGPT** | 관심사/상황 전달, 후보 중 선택 | 프로젝트 컨셉 확정 |
| ② | 요구사항 정의 | **Kiro** | 기능을 자연어로 설명, 검토 & 피드백 | `requirements.md` |
| ③ | 와이어프레임 | **Kiro** | 레이아웃 검토, UI 피드백 | `wireframe.html` (3종) |
| ④ | 기술 설계 | **Kiro** | 아키텍처 검토, 기술 판단 | `design.md`, `api-spec.md` |
| ⑤ | 구현 | **Kiro** | 빌드/실행 확인, 에러 전달 | 소스 코드 (~30 파일) |
| ⑥ | 기능 추가 | **Kiro** | 새 기능 요청, 검토 | 코드 + 문서 업데이트 |
| ⑦ | 브랜딩 | **Gemini** | 시안 선택, 피드백 | 로고, 아이콘, 컬러 팔레트 |
| ⑧ | 문서화 | **Kiro** | 현행화 요청, 검토 | README 3종 + 상세 문서 3종 |

### ① 아이디어 도출 (ChatGPT)

```
나 → ChatGPT: "내 관심사 기반으로 1인 프로젝트 추천해줘"
ChatGPT → 나: 후보 5개 제시
나 → ChatGPT: "트레이딩 쪽으로 더 구체화해줘"
ChatGPT → 나: Hyperliquid DEX 터미널 + AI Copilot 제안
나: ✅ 이걸로 가자
```

산출물: **프로젝트 컨셉** — "Hyperliquid 퍼프 DEX 트레이딩 터미널 + AI Copilot"

### ② 요구사항 정의 (Kiro)

```
나 → Kiro: "포지션 분석 기능을 만들고 싶어. 리스크 점수, 지지/저항, 펀딩, OI, 청산 분석이 필요해"
Kiro → 나: requirements.md 생성 (인수 조건 포함)
나 → Kiro: "오더 분석도 추가해줘. 전략 탐지, 체결 가능성, 집중도, 포지션 영향"
Kiro → 나: 오더 분석 요구사항 추가
나: ✅ 검토 완료
```

산출물:
- `ai-copilot-position-analysis.md` — 포지션 분석 요구사항 (9개 섹션, 인수 조건 포함)
- `ai-copilot-order-analysis.md` — 오더 분석 요구사항 (7개 섹션)

### ③ 와이어프레임 (Kiro)

```
나 → Kiro: "요구사항 기반으로 화면 레이아웃 만들어줘"
Kiro → 나: wireframe.html 생성
나 → Kiro: "빈 상태 화면도 필요해" / "코인 추천 화면도 만들어줘"
Kiro → 나: wireframe-empty.html, wireframe-discover.html 추가
나 → Kiro: "탭 이름 바꿔줘" / "하단 영역 리사이즈 가능하게"
Kiro → 나: 수정 반영
나: ✅ 검토 완료
```

산출물:
- `wireframe.html` — 메인 트레이딩 터미널 (포지션/오더 분석)
- `wireframe-empty.html` — 지갑 미연결 상태
- `wireframe-discover.html` — 코인 디스커버 화면

### ④ 기술 설계 (Kiro)

```
나 → Kiro: "와이어프레임 기반으로 API 설계해줘"
Kiro → 나: api-spec.md 생성 (7개 엔드포인트, Request/Response 스키마)
나 → Kiro: "Rule Engine과 AI를 분리해서 설계해줘"
Kiro → 나: 아키텍처 설계 (Rule Engine + AI Engine 역할 분리)
나 → Kiro: "LLM 폴백 순서 바꿔줘" / "Docker Compose에서 MySQL 빼줘"
Kiro → 나: 설계 수정
나: ✅ 검토 완료
```

산출물:
- `api-spec.md` — REST API 전체 규격 (7개 엔드포인트, 데이터 타입 참조)
- `design.md` — 아키텍처 설계 (데이터 흐름, 모듈 구조)

### ⑤ 구현 (Kiro)

```
나 → Kiro: "설계 기반으로 구현해줘"
Kiro → 나: 파일 단위로 코드 생성 (BE 데이터 레이어 → Rule Engine → AI Engine → API → FE)
나: npx tsc --noEmit → 에러 발생 → 에러 메시지 복붙
Kiro → 나: 에러 수정
나: npm run dev → 런타임 에러 → 에러 메시지 복붙
Kiro → 나: 수정
나: ✅ 빌드 성공, 동작 확인
```

산출물: **전체 소스 코드** (~30 파일, FE + BE)

### ⑥ 기능 추가 — 반복 사이클

구현 이후에도 같은 사이클을 반복하며 기능을 추가했다:

```
"전략 조언(TP/SL) 기능 추가해줘"
  → 요구사항 논의 → 설계 → 구현 → 검토

"코인 추천 기능 만들고 싶어"
  → 아이디어 논의 → 와이어프레임 → Spec 생성 → 구현

"오더 분석에 시장 데이터 패칭 추가해줘"
  → 요구사항 → 구현 → 검토
```

### 핵심 포인트: 검토 사이클

> 모든 단계에서 **AI가 산출물을 만들고 → 내가 검토하고 → 피드백**하는 사이클이 핵심이다.
> AI는 실행자, 나는 의사결정자. 이 역할 분담이 명확해야 품질이 유지된다.

```
┌──────────┐     산출물      ┌──────────┐     피드백      ┌──────────┐
│          │ ──────────────▶ │          │ ──────────────▶ │          │
│  AI 생성 │                 │ 내가 검토│                 │  AI 수정 │
│          │ ◀────────────── │          │ ◀────────────── │          │
└──────────┘     수정본      └──────────┘     승인        └──────────┘
                                                    ──▶ 다음 단계로
```

## 3. 주요 개발 과정 타임라인

### Phase 1: 프로젝트 구조 & 기본 구현

| 단계 | 내용 |
|------|------|
| 프로젝트 셋업 | 모노레포 구조 (web + was), TypeScript, Docker Compose |
| 데이터 레이어 | Hyperliquid API 클라이언트, Redis 캐시, MySQL 연동 |

| 프론트엔드 | Next.js 15, 6개 컴포넌트 (TopBar, Portfolio, Market, AICopilot, Discover, BottomBar) |

### Phase 2: LLM 프로바이더 삽질기

AI 서비스 연동 과정에서 겪은 현실적인 문제들:

| 문제 | 상황 | 해결 |
|------|------|------|
| Gemini 429 에러 | 무료 티어 쿼터가 0 (사실상 사용 불가) | Groq를 1순위로 변경 |
| Groq 모델 폐기 | `llama-3.1-70b-versatile` 모델이 decommission됨 | `llama-3.3-70b-versatile`로 교체 |
| OpenAI 쿼터 초과 | 무료 크레딧 소진 | 3순위 폴백으로 유지 |

최종 폴백 체인: **Groq → Gemini → OpenAI**

교훈: 무료 LLM API는 언제든 제한될 수 있다. 멀티 프로바이더 폴백은 필수.

### Phase 5: 브랜딩 & 비주얼 에셋

코드만 짜는 게 아니라, 프로젝트의 정체성을 만드는 작업도 AI와 함께 진행했다.

#### 프로젝트 네이밍

- ChatGPT와 대화하며 이름 후보를 브레인스토밍
- "Perp DEX Terminal" → 줄여서 **PDEX Terminal**로 확정
- 짧고 기억하기 쉬우면서 도메인(퍼프 DEX)을 담고 있는 이름

#### 로고 & 아이콘 생성

- **Gemini**를 활용하여 로고 시안 생성
- 프롬프트 예시: "미니멀한 트레이딩 터미널 로고, 다크 테마, 블루/퍼플 그라데이션, 차트 모티프"
- 여러 시안 중 마음에 드는 것을 선택하고, 피드백을 반복하며 다듬음
- favicon, OG 이미지 등 필요한 사이즈별 에셋도 Gemini에게 요청

#### 컬러 팔레트 & UI 톤

- AI에게 "다크 테마 트레이딩 터미널에 어울리는 컬러 팔레트 추천해줘"라고 요청
- 배경(#0f0f0f계열), 액센트(블루/그린), 위험 신호(레드/오렌지) 등 역할별 색상 체계 확립
- TailwindCSS 커스텀 컬러로 바로 적용

#### 브랜딩에서 배운 점

- 개발자 혼자서도 AI를 활용하면 "그럴듯한" 브랜딩이 가능
- 로고/이미지 생성은 프롬프트를 구체적으로 쓸수록 결과가 좋음
- 다만 AI 생성 이미지는 세밀한 조정이 어려움 — 완벽보다는 "충분히 좋은" 수준을 목표로

## 4. AI와 협업한 방식의 특징

### 4.1 내가 한 것 vs AI가 한 것

| 나 (의사결정자) | AI (실행자) |
|----------------|-------------|
| 아이디어 제시 및 방향 설정 | 요구사항 문서 구조화 |
| 요구사항/설계 검토 및 피드백 | 와이어프레임 HTML 생성 |
| 기술적 판단 (LLM 선택, 인프라 구성) | API 규격 문서 작성 |
| 빌드/실행 결과 확인 및 에러 전달 | 전체 코드 작성 (FE + BE) |
| UI/UX 피드백 ("탭 이름 바꿔줘") | 코드 수정 및 버그 픽스 |
| 브랜딩 방향 결정 및 시안 선택 | 로고/아이콘 이미지 생성 (Gemini), 컬러 팔레트 제안 |
| 최종 승인 | 문서 현행화 |

### 4.2 효과적이었던 패턴

1. **단계별 검토 사이클**: 한 번에 다 만들지 않고, 요구사항 → 검토 → 설계 → 검토 → 구현 → 검토 순서로 진행. 각 단계에서 방향을 잡아줌.

2. **자연어로 피드백**: "TP/SL 보다는 전략이라는 이름이 좋겠어", "하단 알림 컴포넌트 사이즈 조절할 수 있게 해줘" 같은 일상적인 표현으로 요청.

3. **에러 메시지 복붙**: 빌드 에러나 런타임 에러가 나면 그대로 복사해서 전달. AI가 원인 분석 후 수정.

4. **참고 자료 제공**: 다른 AI(Gemini)에게 받은 답변을 보여주며 "이런 느낌으로 만들어줘"라고 요청. AI가 해당 형식을 분석해서 구현.

### 4.3 주의할 점

- AI가 만든 코드를 무조건 신뢰하면 안 됨. 빌드/실행 확인은 필수.
- 문서와 코드가 따로 놀 수 있음. 구현 후 문서 현행화를 꼭 해야 함.
- LLM API 제한, 모델 폐기 등 외부 의존성 이슈는 직접 해결해야 함.
- 큰 기능은 한 번에 요청하지 말고 작은 단위로 나눠서 진행하는 게 효과적.
